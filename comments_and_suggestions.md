# Предложения по улучшению документации ReFashion

## Критические проблемы

### 1. JSON для `desired_categories` и `desired_brands` в `items`
**Проблема:** JSON-поля не индексируются, нельзя сделать FK-ограничения, нельзя нормально фильтровать по ним ("покажи вещи, которые хотят обменяться на юбку").

**Решение:**
- Выделить таблицу `categories (category_id, name, slug)` — единый справочник категорий
- Поле `category` в `items` → FK на `categories.category_id`
- Убрать `desired_categories` JSON → создать junction-таблицу `item_desired_categories (item_id, category_id)`
- Аналогично `desired_brands` → таблица `brands (brand_id, name)` + `item_desired_brands (item_id, brand_id)`

---

### 2. Дублирование хранения изображений
**Проблема:** В таблице `items` есть `main_image_url` + `images_urls` (JSON), И при этом есть отдельная таблица `item_images`. Это два конкурирующих способа хранения одних и тех же данных.

**Решение:** Убрать `main_image_url` и `images_urls` из `items`. Добавить в `item_images` поле `is_main BOOLEAN` или `display_order INT` для обозначения главного фото.

---

### 3. Противоречие: сессии в Redis vs таблица `sessions` в PostgreSQL
**Проблема:** Архитектура говорит "Redis — хранение активных сессий", но есть полноценная таблица `sessions` в PostgreSQL. Неясно, что является источником истины.

**Решение:** Определить одно место:
- Если используется JWT — сессии вообще не нужно хранить в PostgreSQL
- Если сессионные токены — Redis достаточно, PostgreSQL-таблица избыточна
- Если нужна история сессий (для безопасности) — хранить в PostgreSQL, Redis использовать только как кэш для быстрой проверки

---

## Значительные проблемы

### 4. Неверная кардинальность в `reviews`
**Проблема:** В разделе связей указано `Exchanges.exchange_id → Reviews.exchange_id` как **1:1**, но оба участника обмена должны оставить отзыв друг на друга. Таблица уже содержит `reviewer_id` + `reviewed_user_id`, что подразумевает 2 отзыва на обмен.

**Решение:** Изменить на **1:N**, добавить уникальный составной индекс `UNIQUE(exchange_id, reviewer_id)`.

---

### 5. `payment_status` дублируется в `subscriptions`
**Проблема:** В таблице `subscriptions` есть `payment_status ENUM('pending', 'completed', 'failed', 'refunded')`, хотя для этого есть таблица `payment_transactions`. Данные будут рассинхронизироваться.

**Решение:** Убрать `payment_status` из `subscriptions`. Статус подписки — это `is_active` + даты. Статус платежа — это `payment_transactions.status`.

---

### 6. `auth_method` как ENUM ограничивает пользователя одним методом входа
**Проблема:** Если пользователь зарегистрировался через email, он не сможет привязать Госуслуги к тому же аккаунту.

**Решение:** Вынести в отдельную таблицу `user_auth_providers (provider_id, user_id, provider ENUM('email','gosuslugi'), external_id, created_at)`. В `users` убрать `auth_method` и `gosuslugi_id`.

---

### 7. `messages.read_at` не учитывает двух участников
**Проблема:** Одно поле `read_at` — но чат двусторонний. Для отправителя сообщение всегда "прочитано".

**Решение:** Либо хранить `read_at` только для не-отправителя (задокументировать это явно), либо создать таблицу `message_reads (message_id, user_id, read_at)`.

---

## Менее критичные проблемы

### 8. Неоднозначность статусов обмена
`ENUM('proposed', 'accepted', 'rejected', 'in_progress', 'completed', 'cancelled')` — в документации процессов нет перехода в `in_progress`, есть только `proposed → accepted → completed`. Что означает `in_progress`? Нужно задокументировать state machine переходов.

---

### 9. Нет механизма мягкого удаления (soft delete)
При удалении пользователя или вещи история обменов и отзывов разрушится. Рекомендуется добавить `deleted_at TIMESTAMP` в `users` и `items`.

---

### 10. `rating` в `users` — денормализованное поле
Это кэшированное значение, которое нужно пересчитывать при каждом новом отзыве. Нужно явно задокументировать, что оно вычисляемое и как обновляется, либо всегда считать `AVG(rating)` из таблицы `reviews` динамически.

---

### 11. Опечатка в `payment_method` ENUM
```sql
-- Текущий вариант (ошибка):
ENUM('card', sbp, 'sberbank')

-- Правильный вариант:
ENUM('card', 'sbp', 'sberbank')
```

---

### 12. `UUID/INT` — нужно выбрать одно
Все ID типизированы как "UUID/INT". Это нужно зафиксировать до разработки — выбор влияет на производительность, шардирование и API.

---

### 13. Нет таблицы `notifications`
В описании процессов упоминаются уведомления ("User1 получает уведомление"), но таблица в схеме отсутствует. Нужно либо добавить таблицу, либо явно описать способ доставки (push/email/SSE без хранения).

---

### 14. `last_message_date` в `chats` — денормализация без оговорки
Это поле может рассинхронизироваться с реальными данными `messages`. Нужно документировать как кэш и описать логику обновления.

---

### 15. `condition` не упомянут в форме добавления вещи
В описании формы есть поле "Описание (состояние, дефекты)", но нет отдельного поля для `condition ENUM`. При этом в таблице `items` это обязательное поле. Нужно явно добавить его в описание формы.

---
